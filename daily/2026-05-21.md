## 学习内容

### RAG 章节核心要点

- [ ] **RAG ≠ 接向量库**：RAG 是一条证据链：取回 → 筛选 → 引用 → 交给模型生成
- [ ] **Chunking**：按结构切分（标题/API/函数），不要按固定字数切。每个 chunk 带来源 URL、标题路径、更新时间、版本
- [ ] **Vector DB**：存 embedding + metadata（来源/版本/可信等级/是否废弃）。向量相似 ≠ 答案正确
- [ ] **Retriever**：不止语义相似度。用户问哪个版本？哪个链？官方还是社区？版本/环境/地址要进过滤
- [ ] **Rerank**：初检后重新排序，减少"标题相似内容不对"的问题。资产/开发者文档场景值得加
- [ ] **Citation**：答案要能追溯到具体来源。区分：来自文档 / 模型归纳 / 证据不足 / 需要核对版本
- [ ] **拒答能力**：找不到证据时应该说"不确定"，别让模型补全

### 最小实践：协议文档 RAG 问答

> 选一个官方文档站，抓 10-20 篇页面，按标题结构切 chunk 存向量库。
> 测试三类问题：
> 1. 文档中存在的 API 用法
> 2. 文档中不存在的问题 → 要求拒答
> 3. 旧版本/不确定版本 → 提示需要核对版本
>
> 输出格式：answer / sources / uncertainties / needs_version_check

### AI × Web3 关联

- RAG 位在 Knowledge Base 和模型之间，帮 Agent 查资料、补上下文、引用证据
- 当 RAG 结果影响链上动作时，还需要 simulation + policy + human check
- "文档说可以调用" ≠ "当前用户应该签名"

### MCP 章节精读

> 昨天开篇认识了 MCP 是"AI 客户端和工具之间的通用语言"，
> 今天深入理解 MCP 的架构、协议细节，以及为什么它是你工具链的"底座"。

- [ ] **MCP 架构三要素**：Client → Server → Capabilities
  - **Client**：发起请求的一方（你的 Agent / Hermes / Claude Desktop）
  - **Server**：提供能力的一方（文件系统、数据库、API、链上 RPC）
  - **Capabilities** 分为三类：
    - **Tools**：可执行的操作（搜文件、查数据库、调合约）
    - **Resources**：可读取的数据（文档、配置、状态）
    - **Prompts**：可复用的提示模板
- [ ] **Transport 层**：MCP 不绑传输方式
  - stdio：本地进程通信（你现在的 Hermes 工具链就在用）
  - HTTP + SSE：远程服务通信
- [ ] **为什么 MCP 重要（对你）**：
  - 你每天在用 Hermes Agent，它的工具调用底层就是 MCP 协议
  - 理解 MCP = 理解你的 Agent"手"是怎么伸出去的
- [ ] **MCP vs 传统 API**：
  ```
  传统 API：每个工具各自定义接口 → 需要为每个工具写适配代码
  MCP：统一协议 → 任何 MCP Client 可以接任何 MCP Server
  ```
- [ ] **与 Web3 的关联**：
  - 链上 RPC、合约调用、钱包签名都可以封装成 MCP Server
  - 未来你的 Agent 可能通过 MCP 直接操作链上资产
  - 安全性核心：MCP Server 的权限边界 = 你的资产安全边界

### Vibe Coding 快速回顾

> Gustave 已经有 Vibe Coding 经验（用 Claude Code 做过网页工具），
> 这个章节不是"从零学"，而是"把经验串成体系"。

- [ ] **Vibe Coding 是什么**：用自然语言描述需求 → AI 生成代码 → 你审查和迭代
  - 不是"AI 替你写代码"，而是"你和 AI 结对编程"
- [ ] **你已经掌握的核心能力**（对照你自己的经验）：
  - ✅ 用自然语言描述功能需求
  - ✅ 审查 AI 生成的代码是否合理
  - ✅ 迭代修改：不满意 → 改描述 → 重新生成
- [ ] **Vibe Coding 的边界**：
  - 适合：原型、小工具、UI、CRUD
  - 不适合：安全敏感代码（私钥管理、签名逻辑）、复杂业务逻辑
  - 你需要在 Web3 场景中格外小心：不靠 Vibe Coding 写合约核心逻辑
- [ ] **从 Vibe Coding 到独立开发的路径**：
  - 现在：能描述 + 能审查 + 能迭代
  - 下一步：能读懂框架代码结构 + 能定位 bug
  - 再下一步：能独立写模块

### Phase 1 回顾：AI 基础知识点串联

概念关系图：

```
你（用户）
  ↓ 提问
LLM（大脑） → 理解意图
  ↓ 调用 Prompt（指令工程）→ 引导模型行为
  ↓ 受限于 Context（上下文窗口）→ 决定能记住多少
  ↓ 超出上下文 → RAG（外挂知识库）→ 查资料补上下文
  ↓ 需要行动 → Agent（执行循环）→ 思考/工具/反馈闭环
  ↓ 组织复杂逻辑 → Frameworks（框架）→ 编排流程
  ↓ 连接外部工具 → MCP（协议）→ 标准化工具接入
  ↓ 最终产物 → Vibe Coding（用 AI 写代码）
```

每个概念不是在学"另一个东西"，而是在学"AI 系统的一个零件"。你正在从拼零件变成懂系统。

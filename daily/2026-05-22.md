## 学习内容

### Agent 章节核心要点

- [ ] **Agent 不是"自主权"，而是"被约束的执行循环"**：目标、工具、状态、权限、停止条件缺一不可
- [ ] **Tool Use**：工具让 Agent 从"会回答"变成"能做事"——但也让风险从"答错"变成"做错"
  - 工具设计必须明确：输入 schema / 权限范围 / 只读 or 写 / 副作用 / 调用记录 / 人工确认
- [ ] **Planning**：计划是候选路线，不是授权。好计划能暴露：每步用什么工具、读写类型、自动/手动、失败重试、验证方式
- [ ] **State**：state 必须外置和可查——不能只藏在模型上下文里。还要记录环境、版本、参数、调用结果、确认、撤销
- [ ] **Reflection**：自我检查提高质量，但只能辅助诊断——写入/授权/支付不能依赖 reflection 做最终判断
- [ ] **Multi-Agent**：分工协作放大协调问题——上下文丢失、责任模糊、错误传播、权限扩散。先问：多个 Agent 真减少了复杂度吗？

### 关键概念速记

> **模型提出候选行动 → 系统限制行动空间 → 用户批准高风险边界。**
> 最危险的设计：模糊目标 + 广泛工具 + 长期记忆 + 大额资产权限。

### AI × Web3 安全架构（稳的 Agent 架构）

```
用户提出目标和约束
  → Agent 读取上下文并生成计划
  → 系统拆为只读步骤 + 候选写入步骤
  → 只读自动执行，写入进入 policy 检查
  → Simulation 展示链上影响
  → 用户确认高风险动作
  → Wallet / Smart Account 执行
  → 日志记录每一步和最终状态
```

### 最小实践：DAO 提案研究 Agent

> **只读版本：**
> 1. 读提案正文
> 2. 检索论坛讨论
> 3. 总结支持和反对理由
> 4. 标记缺失信息
> 5. 输出投票前检查清单（不要直接投票！）
>
> **输出要求：**
> - 用了哪些来源
> - 哪些结论证据不足
> - 是否发现治理或资金风险
> - 如果用户要投票，还需要人工检查什么
>
> **权限升级版（后续）：** 只有用户明确授权 + 投票交易 simulation 后才允许生成投票交易草稿。

## 学习内容

### Frameworks 章节核心要点

> 框架解决的是"谁调谁、怎么传、出错怎么办"的编排问题——不是魔法。

- [ ] **框架解决什么**：把 LLM 调用、工具选择、状态管理、错误处理、重试逻辑串成一个可维护的流程。没有框架 = 自己写这些 glue code。
- [ ] **LangChain**：链式调用 + 工具集成。快速原型很强，但复杂场景下调用链容易失控。适合"先跑起来看看"。
- [ ] **LangGraph**：有向图 + 状态机模式。每一步明确输入/输出/分支条件。适合有分支、有回路、有暂停确认的 Agent 工作流。
- [ ] **Agents SDK (OpenAI)**：围绕 Agent Loop 设计的官方框架：handoff、guardrails、tracing。和 OpenAI 生态结合紧密。
- [ ] **框架 ≠ 安全**：框架帮你组织代码，不保证输出安全。框架里的工具权限、资产访问、Prompt 注入防护仍是你的责任。

### 如何选择框架

```
快速原型 / 直线流程     → LangChain
有分支 / 循环 / 确认点  → LangGraph
OpenAI 生态 / 快速部署  → Agents SDK
已有 MCP 工具链         → 直接调 MCP client
```

### MCP 章节开篇

- [ ] **MCP 是什么**：Anthropic 提出的模型-工具-上下文连接协议。让不同的 AI 客户端和不同的工具/数据源能"说同一种语言"。
- [ ] **核心概念**：Client（你的 Agent）→ Server（你的工具/数据源）→ Tools / Resources / Prompts 三类能力
- [ ] **为什么重要**：你已经有 Hermes Agent 在用 MCP — 理解它不是在学"另一个框架"，而是在理解你现在工具链的底层协议。
- [ ] **和 Web3 的关系**：未来 Web3 工具（RPC、合约调用、钱包签名）可以标准化为 MCP Server，让任意 Agent 通过协议接入链上操作。

**验收标准：**
- 模型说"我需要查资料" → 调用 search tool → 拿到结果后再回答
- 最多 3 轮循环后强制停止（safety stop）
- 不需要任何 Web3 知识，用 Python print 代替工具调用

### AI × Web3 关联

- Web3 Agent = AI 框架 + Web3 工具 + 权限边界
- 框架选型影响你的 Agent 的可恢复性、审计能力和安全边界
- MCP 可能是未来"让任意 Agent 安全接入链上"的标准化入口
